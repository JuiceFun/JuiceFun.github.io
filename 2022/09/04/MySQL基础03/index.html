1.三种删除

  1. delete

删除表中的数据，属于DML语句；
表中 的数据被删除，但是物理磁盘上的空间并没有被释放；
优点：支持回滚；
缺点：删除效率低；
语法：delete from 表名；

  1. trancate

删除效率高，表被一次截断，物理删除；
优点：删除效率高；
缺点：不支持回滚；
语法： trancate table 表名;

当表非常大的时候，使用delete会很慢，但是trancate快的一比；

  1. drop

删除的是表，不是表中的数据；
语法：drop table表名；

2.对表结构的修改

create,drop,alter 属于DDL语句
实际中一般很少这样操作；

3.约束

1.什么是约束（constraint）

创建表的时候可以对表的字段加上一些约束，保证表中数据的完整性，有效性；
换言之，约束的作用：保证数据的有效性；
常见约束：

1.非空约束 not null
2.唯一性约束 unique
3.主键约束 primary key
4.外键约束 foreign key

2.非空约束

规定字段的值不可以为NULL。

3.唯一性约束

表示该字段的值不可以重复；

多个字段联合unique

如果一个字段既unique也not null，那么该字段就是主键；

4.主键约束

主键的特征：unique+not null；

  • 任何一张表都必须有主键；
  • 主键是某一行数据的唯一标识；
  • 主键唯一，但是可以复合主键；
    • 复合主键： primary key(id,age)// id和age一起作为主键；
    • 不建议使用复合主键；
  • 主键一般使用：int,bigint ,char,等定长类型；
  • 一般使用自然主键多一点，业务挂勾修改起来麻烦；
  • 一种机制可以自动维护主键：
    • create table( id int primary key auto_increment )//表示从1开始自增

4.存储引擎

存储引擎指的是表存储、组织数据的方式。

  • MySQL默认的存储引擎是InnoDB;
  • MySQL默认字符编码方式是utf8;
  • 可以在创建表的时候指定存储引擎

InnoDB存储引擎
这是mysql默认的存储引擎，同时也是一个重量级的存储引擎。
InnoDB支持事务，支持数据库崩溃后自动恢复机制。
InnoDB存储引擎最主要的特点是：非常安全。

它管理的表具有下列主要特征：
    – 每个 InnoDB 表在数据库目录中以.frm 格式文件表示
    – InnoDB 表空间 tablespace 被用于存储表的内容（表空间是一个逻辑名称。表空间存储数据+索引。）

    – 提供一组用来记录事务性活动的日志文件
    – 用 COMMIT(提交)、SAVEPOINT 及ROLLBACK(回滚)支持事务处理
    – 提供全 ACID 兼容
    – 在 MySQL 服务器崩溃后提供自动恢复
    – 多版本（MVCC）和行级锁定
    – 支持外键及引用的完整性，包括级联删除和更新

InnoDB最大的特点就是支持事务：
    以保证数据的安全。效率不是很高，并且也不能压缩，不能转换为只读，
    不能很好的节省存储空间。

5.事务

1.什么是事务？
事务就是一个完整的业务逻辑，是一个最小工作单元，不可再分；

场景：银行系统中，账户A向账户B转账10000元。
step1:把A账户减少10000；
step2:把B账户减少10000；

step1和step2整体是一个最小工作单元，要么同时成功要么同时失败；
所以事务就是批量的DML语句同时执行成功或者同时执行失败。

2.只有DML语句（update,insert,delete）才有事务这么一说，其它语句和事务无关。

3.事务是怎么做到多条DML语句同时成功，或者同时失败的呢？
InnDB存储引擎提供一组记录事务性活动的日志文件；

事务开启了
insert
insert
update
update
事务结束了

在执行事务的过程中，每一跳DML语句操作都会记录到事务性活动的日志文件中；
在执行事务的过程中，可以提交或者回滚事务。

4.提交和回滚
提交事务commit:

清空事务性活动日志文件,将数据全部持久化到数据库表中;
提交事务意味着事务的结束,是执行成功的结束;

回滚rollback

将之前所有的DML操作撤销,清空事务性活动日志文件;
回滚事务意味着事务的结束,是执行失败的结束;

回滚只可以回滚到上一次提交点.
事务对应的单词:transaction

  • MySQL中默认的是每一条DML语句自动提交一次;
  • 关闭自动提交机制: start transaction;

5.事务的四个特性

  1. 原子性

事务是最小的工作单元，不可再分；

  1. 一致性

在同一个事务中，所有操作要么同时成功，要么同时失败；
保证了数据的一致性

  1. 隔离性
  2. 持久性

事务提交，相当于数据同步到磁盘中。

隔离性

  • read uncommitted
    • A事务可以读取到B事务没有提交的数据（脏数据）；
    • 会导致脏读问题，如A读取了脏数据，B一个回滚……
    • 这是最低隔离级别，一般不会有数据库使用这种；
  • read committed
    • A事务可以读取到B事务已经提交了的数据；
    • 会造成不可重复读
      • A事务读取了B提交的数据，和A一开始启动时读取的不一样了；
    • 这种隔离级别读取到的数据是真实的；
  • repeatable read（MySQL默认隔离级别）
    • B事务已经提交了的数据，A事务也读取不到，A只可以读取到A事务刚启动时的数据；
      • A事务启动时做了备份，B修改数据就不会影响；
      • 类似于银行总账，一条总账的操作可能执行几个小时，这期间如果有人存取，不会影响；
    • 可能存在幻读；
      • 幻读：A事务修改表中的全部数据（如把学生成绩由分数改为等级），这个过程中，B事务插入了一个学生的成绩，A事务完成之后，发现还有一条为处理，这就是幻读；最高隔离级别可以解决此问题。
    • 每一次读到的数据都是幻象，不是真实的。
  • serializable
    • 解决了所有问题，事务必须排队，不允许并发；
    • 每一次读到的数据都是真实的，效率也是最低的；
