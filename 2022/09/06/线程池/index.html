2.构造方法

a. 整体策略与流程：

  1. 线程池一开始没有线程，有任务来时才创建核心线程；
  2. 任务刚来先优先交给核心线程处理；
  3. 如果核心线程达到coreSize并且全部正忙，会先放到阻塞队列中；
  4. 如果核心线程正忙，阻塞队列也满了，会创建救急线程来执行；
  5. 当救急线程也忙，才执行拒绝策略；
  6. 生命周期：
    1. 核心线程，执行完了会保留；
    2. 救急线程执行完了，如果一段时间没有任务会销毁，具体时间是keepAliveTime和timeUnit参数，需要时再创建。

注意：救急线程存在的前提是使用的是有界队列，如果是无界队列，没有救急线程；

b.拒绝策略：
JDK的拒绝策略有四种：

  • AbortPolicy： 让调用者抛出 RejectedExecutionException 异常，这是默认策略；
  • **CallerRunsPolicy： **让调用者运行任务；
  • DiscardPolicy：放弃本次任务；
  • DiscardOldestPolicy： 放弃队列中最早的任务，本任务取而代之。

其它著名策略：

  • Dubbo 的实现，在抛出 RejectedExecutionException 异常之前会记录日志，并 dump 线程栈信息，方

便定位问题；

  • Netty 的实现，是创建一个新线程来执行任务；
  • ActiveMQ 的实现，带超时等待（60s）尝试放入队列，类似我们之前自定义的拒绝策略；
  • PinPoint 的实现，它使用了一个拒绝策略链，会逐一尝试策略链中每种拒绝策略。

3.FixedThreadPool

固定大小的线程池。

  • 核心线程数==最大线程数，没有救急线程被创建；
  • 阻塞队列是无界的队列；
  • 适用于任务量已知，相对耗时的任务；

4. CachedThreadPool

  • 核心线程数是 0， 最大线程数是 Integer.MAX_VALUE，救急线程的空闲生存时间是 60s,意味着：
    • 所有线程都是救急线程；
    • 救急线程可以被无限创建；
  • 队列采用了 SynchronousQueue 实现特点是，它没有容量，没有线程来取是放不进去的（一手交钱、一手交货）。

5.SingleThreadExecutor

  • 单线程线程池，只有一个线程，任务排队执行，任务队列是无界队列；
  • 与单个线程的区别：
    • 单个线程如果任务执行失败而终止那么没有任何补救措施，单线程线程池，失败了会再创建一个线程执行；
  • 与Executors.newFixedThreadPool(1) 的区别：
    • Executors.newSingleThreadExecutor() 线程个数始终为1，不能修改，
      • 对返回值做了封装：FinalizableDelegatedExecutorService 应用的是装饰器模式，只对外暴露了 ExecutorService 接口，因此不能调用 ThreadPoolExecutor 中特有的方法；
    • Executors.newFixedThreadPool(1) 初始时为1，以后还可以修改：
      • 对外暴露的是 ThreadPoolExecutor 对象，可以强转后调用 setCorePoolSize 等方法进行修改。

7.关闭线程池

  • void shutdown():线程池状态变为SHUTDOWN；
    • 不接受新的任务；
    • 已经提交的任务执行结束；
    • 此方法不会阻塞调用线程的执行。
  • List<Runnable> shutdownNow()
    • 不会接受新的任务；
    • 返回任务队列中的任务；
    • 并用 interrupt 的方式中断正在执行的任务。
  • boolean isShutdown():不在 RUNNING 状态的线程池，此方法就返回 true；
  • boolean isTerminated():线程池状态是否是 TERMINATED;
  • boolean awaitTermination(long timeout, TimeUnit unit) throws InterruptedException:调用 shutdown 后，不会等待所有任务运行结束，因此如果它想在线程池 TERMINATED 后做些事情，可以利用此方法等待。

10. Fork/Join

  • Fork/Join 是 JDK 1.7 加入的新的线程池实现，它体现的是一种分治思想，适用于能够进行任务拆分的 cpu 密集型运算；
  • 所谓的任务拆分，是将一个大任务拆分为算法上相同的小任务，直至不能拆分可以直接求解。跟递归相关的一些计算，如归并排序、斐波那契数列、都可以用分治思想进行求解；
  • Fork/Join 在分治的基础上加入了多线程，可以把每个任务的分解和合并交给不同的线程来完成，进一步提升了运算效率；
  • Fork/Join 默认会创建与 cpu 核心数大小相同的线程池；
  • 提交给 Fork/Join 线程池的任务需要继承 RecursiveTask（有返回值）或 RecursiveAction（没有返回值）。
